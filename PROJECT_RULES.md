# プロジェクト全体のルール・制約

この文書は、「Web収益だけで株を買う日記」で作業するときに最初に確認する共通ルールです。記事固有の文章・レイアウトは `CONTENT_STYLE_GUIDE.md`、現在の進捗は `HANDOFF.md`、詳細仕様と変更履歴は `spec.md` を参照してください。

## 1. プロジェクトの目的

- 大学生がAIとWebを使い、0円から収益を作って投資を学ぶ過程を公開する。
- サイトは投資助言ではなく、個人の学習・運営記録として扱う。
- 成功だけでなく、迷い、未達、失敗、収益や損失も記録する。
- 最初の目標は「Webで1円を作る」とする。

## 2. お金と投資の絶対ルール

- 生活費や貯金を投資に使わない。
- 投資に使えるのは、このサイトを含むWeb運営から実際に発生した収益だけとする。
- 借金、クレジットカードの分割払い、無理な支払いを使わない。
- 信用取引は行わない。実取引を検討するときも現物・少額を前提とする。
- Web収益が発生するまでは実取引へ進まない。
- 勝敗や都合にかかわらず結果を記録する。

## 3. 表現と情報の制約

- 「この株を買うべき」「今が狙い目」「確実に上がる」など、推奨・利益保証・売買誘導に見える表現を使わない。
- 「必ず」「絶対」「誰でも」「簡単に稼げる」「知らないと損」などの断定や煽りを使わない。
- 利益だけでなく、損失、元本割れ、価格変動、費用、制度変更の可能性も示す。
- 実際に経験していない申込み、取引、収益、損失、AI利用結果を創作しない。
- 事実、一般的な説明、個人の体験・意見を区別する。
- 制度、税制、NISA、料金、サービス仕様など変わり得る情報は、公開前に公式情報で確認する。
- 変わり得る情報には公式ページへのリンクと情報確認日を付ける。
- 投資記事には、投資助言ではないこと、個人の学習記録であること、最新情報を公式サイトで確認することを明記する。

## 4. 記事制作のルール

- 文体は「です・ます調」、運営者の一人称は「私」で統一する。
- 専門家として教え切る口調を避け、「確認した」「整理した」「まだ分からない」を基本にする。
- 記事構成、本文量、会話、クイズ、記事末尾の順番は `CONTENT_STYLE_GUIDE.md` に従う。
- HTML内へ新しい色コードや `style` 属性を追加せず、既存のCSS変数とクラスを使う。
- 新しい装飾、グラデーション、発光、カード、アニメーションを安易に増やさない。
- 主となる次ページ導線は1個、関連記事は最大3件とする。
- 個人を特定できる情報や、投資と無関係な詳しい属性を公開しない。

## 5. 広告・収益化のルール

- 読まれる土台と記録を優先し、広告枠をむやみに増やさない。
- PR、広告、アフィリエイトを含む場合は、記事冒頭と対象リンク付近に文字で明記する。
- 投資系アフィリエイトは投資助言に見えやすいため、特に慎重に扱う。
- 広告や提携の有無で評価や記録内容を変えない。

## 6. 実装・更新時のルール

- 変更前に関連ページと既存の共通パターンを確認する。
- 新規記事を追加したら、少なくとも次を同期する。
  - `index.html` の記事カードと公開記事数
  - `roadmap.html` の該当項目
  - 前後記事のナビゲーション
  - `sitemap.xml`
  - `spec.md` のファイルマップと変更履歴
  - `HANDOFF.md` の進捗
- 共通ナビ、フッター、計測タグ、SEOメタ情報を変更するときは全対象ページを確認する。
- 公開前にリンク切れ、HTML構造、モバイル表示、キーボード操作、横スクロール、モーション抑制設定を確認する。
- 制作メモや内部情報をGitHub Pagesの公開成果物へ含めない。
- ユーザーの既存変更を無断で取り消さない。

## 7. 完了条件

作業は、ファイルを変更しただけでは完了としません。次を満たして完了とします。

- 要求した内容が実装されている。
- 関連ページとドキュメントが同期されている。
- 可能な範囲で表示・リンク・構文を確認している。
- 未確認事項や既知の問題があれば `FAILURE_LOG.md` または `HANDOFF.md` に残している。

## 8. 文書の優先順位

ルールが重複・矛盾した場合は、次の順で確認します。

1. 作業時にユーザーから受けた最新の指示
2. `PROJECT_RULES.md`
3. `CONTENT_STYLE_GUIDE.md`
4. `spec.md`
5. `HANDOFF.md`
6. `blog/operations.md`

矛盾を発見した場合は、勝手に解釈して複数文書へ広げず、影響を確認してから文書を同期します。
